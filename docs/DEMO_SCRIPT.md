# PRD Copilot 演示脚本

## 30 秒版

`PRD Copilot` 是一个面向产品经理的需求文档与流程图生成助手。  
用户输入一句自然语言需求后，系统会先拆成结构化字段，再生成固定 8 个模块的产品文档，同时支持风格改写、质量检查、历史版本记录和 `.drawio` 流程图导出。  
这版仓库支持 `Demo Mode` 和 `Live Mode`，求职展示时我默认用 `Demo Mode` 保证稳定性，`Live Mode` 用来补充真实模型接入能力。

## 2 分钟版

### 1. 问题是什么

- 产品经理写 PRD 和功能说明时，原始信息通常来自会议记录、口头需求和零散笔记。
- 从零散信息到结构化文档、再到流程图表达，整理成本很高。
- 文档评审前还会反复改写，不同协作对象看到的信息粒度也不一致。

### 2. 产品怎么设计

- 用户先输入一句需求或一段需求描述。
- 系统先做结构化拆解，而不是直接输出长文，提升稳定性和可控性。
- 文档固定生成 8 个模块，包括背景、痛点、目标、功能、流程、页面和指标。
- 流程图不让模型直接输出 `.drawio`，而是先生成结构化节点，再由程序转成文件。
- 结果页支持改写风格、质量检查、协作视图和历史版本回看。

### 3. 为什么这版适合演示

- `Demo Mode` 可以稳定跑通全链路，适合简历和面试演示。
- `Live Mode` 保留真实模型接入，用来说明技术完整度。
- 仓库本身已经是独立项目结构，适合直接开源展示。

## 演示顺序建议

### 30 秒快演示

1. 首页输入一句需求
2. 保持 `Demo Mode`
3. 点击“一键完整生成”
4. 依次切换文档、流程图、质量检查、版本

### 2 分钟标准演示

1. 先说明用户是谁、问题是什么
2. 展示一句需求如何被拆成结构化字段
3. 展示固定 8 段文档输出
4. 点一个改写风格，演示版本变化
5. 切到流程图页，说明 `.drawio` 由程序生成
6. 切到质量检查、协作视图和历史记录页

### Live Mode 补充讲法

如果现场网络允许，再补一句：

- 这版还保留了 `Live Mode`
- 真实模型只负责需求拆解和内容生成
- 严格格式文件仍由程序生成，避免直接依赖模型输出 XML

## 面试展开版

### 为什么这个问题成立

- PRD、功能说明和流程图本来就是产品经理的高频工作。
- 真实工作中输入往往不是一份完整文档，而是碎片化信息。
- 所以这个问题不是“帮用户写字”，而是“帮用户整理需求和统一表达”。

### 为什么适合用大模型

- 大模型擅长自然语言理解、结构化拆解和文档初稿生成。
- 改写风格、补充表述、统一语气也适合由模型完成。
- 但严格格式文件不适合完全交给模型，因此 `.drawio` 由程序生成。

### 你在产品上做了什么决策

- 入口用自然语言输入降低门槛。
- 先拆解、后生成，避免直接长文跑偏。
- 用 `Demo Mode / Live Mode` 区分展示稳定性和技术完整度。
- 结果页不仅有文档，还有质量检查、协作视图和历史记录，强调工具属性而不是单次生成。

### 人工判断为什么还重要

- 缺失项不能瞎补，尤其是目标用户、边界条件和异常流程。
- 质量检查只做提示，不直接覆盖最终文档。
- 最终是否采用某个版本，仍然需要产品经理自己判断。
