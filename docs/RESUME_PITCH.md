# PRD Copilot 简历与面试讲法

## 简历可直接使用的项目描述

`PRD Copilot｜面向产品经理的需求文档与流程图生成助手`

面向产品经理设计需求整理工作台，解决 PRD 撰写中信息零散、结构不统一、表达不完整、流程图整理成本高和版本沉淀分散的问题。  
支持用户通过自然语言输入需求，系统先完成结构化拆解，再生成固定 8 个模块的产品文档，并导出可继续编辑的 `drawio` 流程图。  
项目加入文档改写、质量检查、协作视图、历史记录以及 `Demo Mode / Live Mode` 双模式，用于兼顾求职展示稳定性和真实模型接入能力。

## 简历精简版

如果简历空间更紧，可以压缩成两行：

面向产品经理设计需求文档与流程图生成助手，支持自然语言输入、结构化拆解、文档生成、改写优化和 `drawio` 导出。  
通过 `Demo Mode / Live Mode` 双模式兼顾稳定演示和真实模型接入，并补充质量检查、协作视图和历史版本能力。

## 面试时可以重点讲什么

### 1. 为什么这个问题成立

- 产品经理真实工作里确实需要频繁写 PRD 和功能说明
- 原始需求通常不是完整文档，而是会议记录、口头描述和零散笔记
- 从文字到流程图的转换成本高，评审前还会反复修改

### 2. 为什么适合用大模型

- 大模型适合做需求拆解、文档初稿生成和表达改写
- 但不适合直接输出严格格式文件，所以 `.drawio` 由程序生成
- 质量检查和协作视图则作为模型结果之后的结构化派生能力

### 3. 你在产品设计上做了什么

- 用一句话需求输入降低使用门槛
- 用“先拆解、后生成”的流程提升稳定性
- 用 `Demo Mode / Live Mode` 区分展示稳定性和技术完整度
- 用历史记录和版本视图让项目更像真实工作台，而不是一次性生成器

### 4. 人工为什么还重要

- 缺失项不能瞎补，尤其是目标用户、边界条件和异常流程
- 质量检查只做提示，不直接替代最终判断
- 最终文档是否可用、是否采用，仍然需要产品经理人工确认

## 一句话讲法

这是一个面向产品经理的需求整理工作台，用自然语言输入降低门槛，用结构化生成和程序化导出提高结果可用性，并用 `Demo Mode / Live Mode` 平衡作品展示稳定性和真实模型能力。
